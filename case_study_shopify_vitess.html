Case Study: Horizontally Scaling MySQL at Shopify with Vitess

How Shopify transitioned its massive, high-throughput MySQL database to a horizontally scalable architecture using Vitess to support the exponential growth of the Shop app.

The Challenge: A Victim of Its Own Success

The Shop app's rapid growth pushed its monolithic MySQL database to its limits. The key challenges were:

  • Massive Scale: The database grew to many terabytes, with billions of rows in some tables.
  • Performance Bottlenecks: Schema migrations took weeks, and the database was frequently throttled during peak hours, slowing down background jobs.
  • Limited Scalability: Vertically scaling the single MySQL instance was no longer a viable option, and further splitting the database into more federated databases would add too much complexity to the application layer.

The Architecture: A Phased Migration to Vitess

Shopify adopted a phased approach to migrate to Vitess, a database clustering system for horizontal scaling of MySQL.

graph TD subgraph "Phase 1: Vitessifying" A[Monolithic MySQL] --> B(Vitessified MySQL with VTTablet); B --> C{VTGate Proxy}; end subgraph "Phase 2: Splitting Keyspaces" C --> D[Unsharded 'users' Keyspace]; C --> E[Unsharded 'global' Keyspace]; end subgraph "Phase 3: Sharding" D --> F(Sharded 'users' Keyspace); E --> G(Unsharded 'global' Keyspace); end
  1. Phase 1: "Vitessifying": The existing MySQL database was brought under Vitess management as a single, unsharded keyspace. This involved deploying a `VTTablet` process alongside each `mysqld` process and routing traffic through the `VTGate` proxy. This was done with no downtime.
  2. Phase 2: Splitting into Keyspaces: The tables were then logically separated into different unsharded keyspaces (e.g., `users`, `global`) using Vitess's `MoveTables` workflow. This allowed for better organization and prepared the data for sharding.
  3. Phase 3: Sharding the "Users" Keyspace: The `users` keyspace, which contained the majority of the data, was sharded based on the `user_id`. This involved setting up Sequences for auto-incrementing primary keys and Vindexes for maintaining global uniqueness. The `Reshard` workflow was used to split the data across multiple MySQL shards.

Key Technical Details & Learnings

  • Choosing a Sharding Key: The `user_id` was chosen as the natural sharding key, as most of the data was user-centric. This required a significant effort to backfill the `user_id` on tables where it was missing.
  • Query Verifiers: Shopify built custom query verifiers in the application layer to find and fix queries that would be incompatible with a sharded database (e.g., queries missing a sharding key, cross-shard transactions). This was credited as the single most important factor for a successful migration.
  • Sequences for Auto-Incrementing IDs: To handle auto-incrementing primary keys in a distributed environment, Vitess Sequences were used. These are backed by a MySQL table and managed by the VTTablet to ensure unique, monotonically increasing IDs across all shards.
  • Vindexes for Uniqueness: To enforce uniqueness constraints across shards, Lookup Vindexes were used. These are essentially index tables maintained by Vitess that allow it to check for the existence of a value across all shards before an insert.
  • Phased Rollout and Testing: The entire migration was done with extreme caution, with extensive testing in multiple staging environments. A dynamic connection switcher was built to gradually route traffic to Vitess, minimizing risk.
